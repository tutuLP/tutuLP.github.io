## 考虑区块链

***是否使用区块链的方式？***

应该不行，首先每篇文章的难度难以评定，所以难以根据贡献度发送奖励

***那能否有解决办法？***

首先发布一个主题的教程，可以定一段时间称作一周期，当一周期结束才结算一次奖励，然后定另一个时间成为二周期，二周期结束看在原文基础上是否有更新，根据更新的比例（暂定）划分第二次奖励，并且根据热度，点赞率，阅读时长等发放常驻奖励，如果没有更新则只有常驻奖励，下一个周期成为三周期，每个靠后的周期的奖励可以递减，时间可以拉长等，以一篇主题的文章为维度进行奖励。

***那如何设置区块呢？如何设置节点？***

中心服务节点存储所有内容，全节存储教程的哈希？包含一个主题所有的历史更新？并且矿机只作为算力需求，有计算则必定打包进去，那如何分发请求？随机，设置一个类似tcp的应答机制并且含有超时机制，同时向多个机器发送，最快返回的作为奖励。区块链则需要打包所有“账单”，那如何避免攻击？

> 虽然区块链的方式难以实现，但是奖励机制可以做为参考引入贡献值。

## 核心功能设计

***如何判断文章是否属于同一主题？***

每篇文章的主题和简介都是通过统一标准的AI生成，新的文章会根据主题和简介向量搜索相似度比较高的内容然后交给AI进行判断，如果属于同一主题系统将告知用户已存在相同主题的内容，并且判断如何可以合并给出合并后的内容，如果内容有冲突则作为独立内容发布。

***tag设计思路？***

原本的设计思路是同一主题下通过多标签设计，最大程度实现整合，但是技术实现上复杂度太高，暂时取消此类设计

***内容修复与更新？***

有两种方式争对原有内容进行更新和修改，一是发布完整的内容，修改完成之后需要附带一段修改评论，AI会对内容进行一次判定，如果符合规范会直接替换，但是旧的内容还是会保存在系统中方便后续归档，如果判定有疑虑会暂存，用户访问的时候可以选择较新的文章并进行专门的投票选项，当超过一个比值之后才会替换，一是划线评论，达到一定赞同之后自动AI合入。

***内容保存？***

主题表和内容表相互关联，所有历史文章都得到保存通过当前文章状态区分(use，vote，archive)，用户点赞与收藏记录中的文章是特定的历史版本，某一主题的文章阅读的时候是可以直接查看历史文章的，可以定期清理过时文章。

***文章审核？***

先检测主题，打回或者合并或者新增，同时AI润色和精炼文章，避免废话文学，并进行特定排版

***如何实现文章分类？***

采用固定的三级分类，暂时不使用分类树（自关联ID），主题只关联第三级分类，一二级分类直接使用分类树，第三级分类单独成表，并且需要用一句话描述这个分类，向量编码一级分类，二级分类，三级分类，描述，向量检索之后选取Top10给AI判断往哪个分，如果都不适合AI自动生成新的分类并插入数据库

***内容如何做双语？***

字数少的分类已经主题可以直接在表中存双语，正文内容存储语言以及原文，前端动态翻译，并创建一个缓存表，调用过的文章做一次保存即可。测试一下向量检索是否支持双语匹配，测试GTE-Multilingual-Base，参数是BGE-M3(1024)的一半，768维

## 后端设计

- ID生成

雪花算法-Snowflake有时钟回拨的问题，目前开源的两个Segment号段模式获取ID有美团的Leafhttps://github.com/Meituan-Dianping/Leaf和滴滴的tinyidhttps://github.com/didi/tinyid，都是通过Java实现，但是项目一开始没有必要用，可以使用
Redis INCR，通过INCR获取号段
`INCRBY user:id 10000`  返回10000，表示1-10000，再次申请 `INCRBY article:id 10000` ，返回20000，也就是每10000个ID访问一次，内存中保存号段以及当前号码(原子性)，并且使用双buffer，当第一个号段使用率达到80%的时候，异步申请第二个号段，并采用Redis主从，防止挂掉，将所有业务封装为接口，对外暴露通用方法方便后续迁徙方案

### 计算贡献值

根据发的教程数目，和阅读量，点赞量综合考虑贡献值，根据贡献值获得 title 和金额的奖励

| 行为项                  | 描述                           | 权重    | 限制说明                        |
| ----------------------- | ------------------------------ | ------- | ------------------------------- |
| 发布教程                | 每发布一个原创教程（通过审核） | +100    | 重复发布不加分                  |
| 教程被浏览              | 每次浏览计入                   | +0.1~1  | 设定单IP每天对同一教程最多记1次 |
| 教程被收藏              | 被用户收藏一次                 | +5      | 去重统计                        |
| 教程被点赞              | 被点赞一次                     | +2      | 同一用户只计一次                |
| 教程被评论              | 有质量评论（审核后）           | +3      | 低质评论不计入                  |
| 教程评分                | 教程平均评分高于 4 星          | +10~50  | 每月统计一次                    |
| 教程通过评审上精选/推荐 | 被平台推荐                     | +50~200 | 根据等级给分                    |
| 教程引流注册            | 通过教程页面产生新注册用户     | +20     | 同IP短时间只计1次               |
| 举报有效（帮助平台）    | 举报无效教程成功               | +10     | 仅限首次成功举报                |

月度奖金池：假设平台当月收入为 `R`，拿出 `R × 分红比例（如 30%）` 作为奖金池

### 上传文章

检查

| 检查点             | 工具                                                         |
| ------------------ | ------------------------------------------------------------ |
| 标题层级混乱       | `mistune` / `markdown-it-py`                                 |
| 中文标点/空格/缩进 | 正则或 [textlint 中文规则](https://github.com/textlint/textlint) |
| 拼写检查           | `pyspellchecker` 或 GPT                                      |
| 写作风格分析       | 大模型或 NLP 模块（如 HanLP）                                |

正确性验证，不合理直接打回，可申诉。

高速方案：

本地语法规则 + AST 解析

- 使用 `markdown-it-py` 或 `mistune` 解析 Markdown 结构，形成 AST；
- 写一套 **排版规范规则**，如：
  - 标题层级递增；
  - 中文标题前后空格；
  - 图片/链接语法检查；
- 再输出格式化内容 + 修改建议。

轻量模型替代大模型，可训练 Markdown 格式改写模型。

前端格式化：Textlint或WebAssembly

使用 Textlint + 规则插件

- 使用 [textlint](https://github.com/textlint/textlint)
- 加载规则如：
  - `textlint-rule-preset-ja-spacing`
  - `textlint-rule-no-mix-dearu-desumasu`
  - `textlint-rule-spellcheck-tech-word`

WebAssembly + 本地排版引擎

在浏览器端用 WebAssembly 或 JS 库做本地排版与提示；

- 将 `markdownlint`、`remark` 等 JS 工具封装成 WebWorker；
- Markdown 格式化库：`prettier-markdown`；
- 拼写检查：`typo.js` + `nspell`；
- 将提示通过编辑器界面（如 Toast UI Editor）高亮显示。

生成标题：

1. 先用轻量级模型（如 `text-embedding-3-small` 或 `KeyBERT`）提取段落关键词；
2. 拼接摘要（前段 + 含高权重关键词的句子）；
3. 再喂给 LLM 生成标题。
