# 基础概念

* 什么是产品？
  能解决某个问题的东西就是产品，分为有形的产品和无形的产品。

* 什么是产品经理？
  想清楚产品怎么做的人
* 合格的产品经理需要关注哪些核心问题？
  用户，场景，需求
  功能，体验/优势，价值

* 产品经理分类
  C 端	面向普通用户
  B 端        面相企业或公司内部
  G 端        面相政府部门

  * 目标用户

    核心用户（商业价值）

* * 供需关系
    服务端
    消费端
* * 统计学
    年龄，性别，地域，职业，消费水平
* * 熟练程度
    新用户，老用户，专家用户

* 抓住核心用户：

捆绑式(会员制，限时优惠)，诱导式(低价)，享受型(服务)，网络型(社交，网络社群)，股东型(粉丝)

### 场景化分析

>  感知用户的真实体验感受

* 场景化分析方法
  列举场景要素，构建场景画面，预判发现需求

1. 场景五要素
   用户分析：类型，特征，熟悉程度
   地点：地理位置，使用场所，软件平台(mac，phone，小程序等)
   时间：固定时间，周期性时间(工作日)
   行为：使用流程和操作行为
   目的，用户想要什么

2. 构建画面

上班族在上班通勤的地铁上，通过刷视频来获取最新的新闻资讯

3. 预判用户行为与目标：

组成动作(评论与回复，分析与转发，复制与粘贴等)
常进用户行为(下拉刷新，左滑上一页，双击点赞)
产品使用流程(搜索-筛选-查看详情) 

* 场景化分析的应用
  洞察问题，理解用户，提高预期

  观察
  用户反馈
  问卷调查
  用户访谈

* 一手需求和二手需求
  一手需求，更准确，效率低
  二手需求，效率高，易曲解
  全员收集，统一处理

* 需求优先级

  辨别真实需求，划分需求强烈程度(价值，风险，规模)和需求频率
  优先级：强烈且高频 t0 > 强烈低频 t1 > 不强烈高频 t2

* 用户故事
  将典型用户在应用场景下将需求进行描述的一种方式
  模板：
  故事主题，故事 id，做没<某角色>，需要实现什么目标，原因，验收标准，条件，触发点，预期结果

> 评估一个功能的价值，可以先试点

* 范围价值=潜在用户数*单用户价值
* 频次价值=需求频次*单次价值
* 时间价值=持续时间*单位价值

成本评估
成本=实现时间(天)*每(天)花多少

# MRD

可行性分析(市场，商业，技术，风险)

“**需求从哪里来、为谁解决、解决什么、为什么能成、做成什么样**” 

1. 文档基础信息：明确文档属性与范围

* 关键内容

1. 文档标题（精准指向需求主题，如《2024 年一线城市职场新人通勤租房 APP MRD》）；
2. 版本信息（版本号：V1.0；更新记录：标注迭代时间、修改内容，方便追溯）；
3. 基础信息（编写人 / 团队：如产品部 - 张三；编写日期；文档目标：如 “论证职场新人租房通勤需求的合理性，支撑产品立项”）；
4. 适用范围（如 “公司内部产品、市场、研发、管理层使用，不对外公开”）。

2. 市场背景与趋势：论证 “为什么现在做”

核心是 “阐述需求所处的市场环境”，证明需求并非主观臆断，而是顺应行业趋势、填补市场空白，为后续分析铺垫合理性。

- 关键内容
  1. 行业整体现状（规模、增长：如 “2024 年中国租赁市场规模达 2.8 万亿，年增速 12%，其中‘95 后职场新人’客群占比超 40%”）；
  2. 细分领域痛点（市场未满足的需求：如 “一线城市新人租房时，‘通勤时间’是首要决策因素，但 70% 的租房 APP 仅支持‘区域筛选’，无法精准匹配通勤需求”）；
  3. 驱动因素（政策 / 技术 / 社会趋势：如 “政府‘职住平衡’政策鼓励通勤优化，第三方地图 API 开放通勤数据，技术可实现性提升”）；
  4. 市场空白点（差异化机会：如 “现有产品聚焦‘房源数量’，未针对‘新人通勤效率’做专项优化，存在细分市场机会”）。

3. 目标用户分析：明确 “给谁做”

核心是 “精准定义需求受众”，避免 “泛化用户”，确保所有需求设计围绕 “特定人群的真实场景” 展开，而非 “所有人的模糊需求”。

- 关键内容

  （需场景化、具象化，拒绝抽象描述）：

  1. 用户画像（ demographics + 行为特征 ）：

     - 基础属性：年龄 22-25 岁、刚毕业 1-2 年、月收入 6k-12k、集中在互联网 / 金融行业（一线城市 CBD 工作）；
     - 行为特征：每周看房 2-3 次、依赖 APP 筛选、平均筛选 15 个房源后预约、优先选择地铁通勤（无车）；

  2. 用户分层（核心 / 次要 / 边缘，明确优先级）：

     - 核心用户：一线城市 CBD 无车职场新人（需求优先级最高，是产品的 “基石用户”）；
     - 次要用户：近郊工作、可接受 1.5 小时通勤的新人（需求可后续迭代）；

  3. 用户场景与痛点（

     核心中的核心

     ，用 “场景故事” 替代 “抽象痛点”）：

     - 场景示例：“小李是北京国贸实习生，每天 20:00-22:00 用 APP 找房，想筛选‘地铁 10 号线 30 分钟内到国贸’的房源，但 APP 只能按‘朝阳区’筛选，需手动查每个房源的通勤时间，每次耗时 1.5 小时，常错过优质房源”；
     - 痛点总结：通勤筛选不精准、手动查时间效率低、优质房源易遗漏。

4. 需求分析与优先级：定义 “解决什么问题”

核心是 “将用户痛点转化为明确需求，并排序优先级”，避免后续产品设计 “眉毛胡子一把抓”，确保优先解决 “最核心的痛点”。

- 关键内容

  ：

  1. 需求清单（从用户痛点拆解，需 “用户视角” 而非 “产品视角”）：
     - 需求 1：支持 “自定义通勤时间 + 地铁线路” 筛选房源（对应 “通勤筛选不精准” 痛点）；
     - 需求 2：自动计算房源到目标工作地的通勤时间（对应 “手动查时间效率低” 痛点）；
     - 需求 3：新增 “优质通勤房源实时提醒” 功能（对应 “优质房源易遗漏” 痛点）；
  2. 需求优先级排序（需有明确标准，拒绝主观判断）：
     - 常用方法：KANO 模型（必要 / 期望 / 魅力需求）、RICE 模型（影响范围 / 程度 / 置信度 / 成本）；
     - 示例：需求 1（通勤筛选）是 “必要需求”（无则用户不用）→ 优先级 P0；需求 2（自动计算）是 “期望需求”（有则体验好）→ P1；需求 3（实时提醒）是 “魅力需求”（超出预期）→ P2；
  3. 需求边界（明确 “不做什么”，避免范围蔓延）：
     - 示例：当前不支持 “自驾通勤筛选”（核心用户无车）、不做 “二手房交易”（聚焦租房场景）、不接入 “海外房源”（仅覆盖国内一线城市）。

5。 竞品分析：找到 “差异化机会”

核心是 “分析现有竞品的优劣势，明确自身差异化方向”，避免 “重复造轮子” 或 “盲目竞争”，确保产品在市场中具备独特竞争力。

- 关键内容

  （聚焦 “与用户核心需求相关的功能”，而非全面罗列）：

  1. 竞品选择（2-3 个直接竞品 + 1 个间接竞品）：
     - 直接竞品：贝壳租房（房源多、验真严）、自如 APP（服务标准化）；
     - 间接竞品：58 同城租房板块（覆盖广但非核心业务）；
  2. 竞品对比（围绕 “核心需求” 设计维度，用表格清晰呈现）：
     | 竞品名称 | 通勤筛选功能 | 自动计算通勤时间 | 新人专属服务 | 核心优势 | 核心劣势 |
     |----------|--------------|------------------|--------------|----------|----------|
     | 贝壳租房 | 仅支持 “区域 + 地铁线”，无时间筛选 | 需手动输入工作地，不自动计算 | 无 | 房源多、验真严格 | 通勤筛选不精准，新人无差异化 |
     | 自如 APP | 支持 30/60 分钟两档时间筛选 | 支持，但仅覆盖 50% 地铁线路 | 新人租房礼包 | 装修统一、服务好 | 时间档位粗，线路覆盖不全 |
  3. 差异化机会总结（从竞品劣势提炼）：
     - 机会 1：提供 “自定义通勤时间 + 100% 地铁线路覆盖”（竞品档位粗 / 覆盖不全）；
     - 机会 2：自动计算通勤时间并同步实时地铁客流（竞品无实时数据）；
     - 机会 3：新增 “新人通勤顾问 1 对 1 服务”（竞品无专属服务）。

6. 可行性分析：论证 “我们能不能做”

核心是 “从市场、商业、资源、风险四个维度，论证产品落地的可行性”，是说服管理层立项的关键依据（避免 “需求好但做不了”）。

- 关键内容

  ：

  1. 市场可行性（需求真实 + 市场有空间）：如 “80% 受访新人表示‘通勤筛选’是核心需求，目标细分市场规模达 80 亿 / 年”；
  2. 商业可行性（能赚钱 + 投入产出合理）：如 “商业模式：免费用户看广告，付费会员（19.9 元 / 月）解锁‘优先约看’；初期投入 200 万，首年营收 150 万，第 2 年盈利”；
  3. 资源可行性（现有能力能支撑）：如 “研发团队可复用现有租房 APP 框架，第三方地图 API 可直接接入，无需额外招聘；已对接 10 家中介，房源数据充足”；
  4. 风险与应对（提前预判 + 有解决方案）：如 “风险 1：竞品 3 个月内跟进通勤功能→应对：优先上线‘实时客流 + 顾问服务’，拉大差异化；风险 2：房源数据合规问题→应对：提前与律师确认，签署合规协议”。

7. 产品定位与核心价值：明确 “做成什么样”

核心是 “一句话说清产品的市场位置，以及对用户 / 公司的价值”，确保跨部门团队对 “产品核心” 有统一认知，避免后续方向偏离。

- 关键内容

  ：

  1. 产品定位（精准、易懂）：如 “专为一线城市 22-25 岁无车职场新人打造的租房工具，核心解决‘通勤时间精准匹配’痛点，让找房效率提升 50%”；
  2. 用户价值（用户能获得什么实际好处）：如 “减少 50% 筛选时间、避免因通勤问题换房（节省 2000 元 / 次换房成本）、快速找到‘通勤 + 预算’双达标的房源”；
  3. 商业价值（公司能获得什么回报）：如 “切入 80 亿细分市场，首年获取 100 万用户，后续通过会员、广告、中介分成实现营收，成为‘新人租房’领域标杆产品”。

8. 附录（可选但建议补充）

核心是 “提供辅助信息，增强文档说服力”，内容可根据实际需求增减。

- 常见内容
  1. 调研数据来源（如用户访谈记录、问卷报告、行业白皮书链接）；
  2. 术语定义（如 “通勤时间：指早高峰时段地铁 / 公交的实际耗时，非地图理论耗时”）；
  3. 参考资料（如竞品功能截图、第三方数据报告节选）。

9. 总结：MRD 的 “闭环逻辑链”

完整的 MRD 需形成 “**市场背景→用户痛点→需求定义→差异化机会→可行性论证→产品价值**” 的闭环，最终回答 3 个核心问题：

1. 为什么要做这个产品？（市场有需求、用户有痛点）；
2. 做什么能解决问题？（核心需求 + 差异化功能）；
3. 我们能做成吗？（资源够、风险可控、有商业价值）。

只有覆盖以上 8 大模块，MRD 才能真正成为 “立项依据” 和 “团队共识文档”，为后续立项报告、PRD（产品需求文档）的输出奠定坚实基础。

# 敏捷开发

先将产品所有的需求进行收集，形成需求清单(**backlog**)，给每个需求标注优先级 p0 p1 p2，然后将一个产品的分为不同版本，每个版本实现不同需求(**story**)
sprint阶段：
sprint 1 -> prd -> prereview meeting -> review metting -> plan meeting -> daily meeting -> retro meting -> online -> sprint 2

* prd：从 backlog 中挑选需要实现的story ，完成 prd 文档
* prereview meeting ：和leader 或专家讨论是否写的对或者容易实现
* review metting ：评审 prd
* plan meeting：研发将 完成 story 需要的功能 feature 拆分为多个任务 task，每个 task 需要分到每个研发负责人 owner，并分配完成任务所需工日时间 point，设计完成后可能有的 story 没有实现需要扔回backlog
* 每个研发讲自己 task 进行的到哪里了，能否按时完成，遇到的困难，不能阻塞sprint流程
* retro meting：讨论该 sprint 中遇到的问题，如何改善等
